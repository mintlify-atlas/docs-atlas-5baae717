title: "Outlook Calendar"
description: "Connect your Microsoft Outlook or Office 365 calendar to FluidCalendar for two-way event sync and task scheduling."
---

FluidCalendar integrates with Microsoft Outlook and Office 365 through the Microsoft Graph API using OAuth 2.0. Once connected, your Outlook calendars and tasks are visible in FluidCalendar and stay in sync as you make changes.

## Prerequisites

- A [Microsoft account](https://account.microsoft.com) (personal, work, or school)
- Access to the [Azure Portal](https://portal.azure.com)
- Your FluidCalendar instance running and accessible at a known URL

## Set up an Azure app registration

<Steps>
  <Step title="Register a new application">
    1. Go to the [Azure Portal app registrations page](https://portal.azure.com/#blade/Microsoft_AAD_RegisteredApps/ApplicationsListBlade).
    2. Click **New registration**.
    3. Enter a name for your app (for example, `FluidCalendar`).
    4. Under **Supported account types**, select **Accounts in any organizational directory (Any Azure AD directory — Multitenant) and personal Microsoft accounts (e.g. Skype, Xbox)**.

       <Tip>
         This option allows both personal Microsoft accounts and work or school accounts to connect. If you only want to support accounts from a specific organization, choose the single-tenant option and note your **Tenant ID**.
       </Tip>

    5. Leave **Redirect URI** blank for now — you will add it in the next step.
    6. Click **Register**.
    7. Copy the **Application (client) ID** shown on the overview page — you will need it later.
  </Step>

  <Step title="Configure the platform and redirect URIs">
    1. In your registered app, go to **Authentication**.
    2. Click **Add a platform** and select **Web**.
    3. Under **Redirect URIs**, add:
       - `http://localhost:3000/api/auth/callback/azure-ad` (for local development)
       - `https://your-domain.com/api/auth/callback/azure-ad` (for production)
    4. Under **Implicit grant and hybrid flows**, check both **Access tokens** and **ID tokens**.
    5. Click **Configure**.
  </Step>

  <Step title="Add API permissions">
    1. Go to **API permissions**.
    2. Click **Add a permission → Microsoft Graph → Delegated permissions**.
    3. Search for and select each of the following permissions:

       | Permission | Purpose |
       |---|---|
       | `Calendars.ReadWrite` | Read and write calendar events |
       | `Tasks.ReadWrite` | Sync Outlook Tasks |
       | `User.Read` | Read the signed-in user's profile |
       | `offline_access` | Maintain access using a refresh token |
       | `openid` | OpenID Connect sign-in |
       | `profile` | Read the user's basic profile |
       | `email` | Read the user's email address |

    4. Click **Add permissions**.
    5. If you are an Azure administrator, click **Grant admin consent for [your organization]** to pre-approve these permissions for all users. Otherwise, each user will be prompted to consent when they connect their account.

    <Note>
      The `offline_access` permission is required for token refresh. Without it, users would need to reconnect their account every hour.
    </Note>
  </Step>

  <Step title="Create a client secret">
    1. Go to **Certificates & secrets**.
    2. Click **New client secret**.
    3. Enter a description (for example, `FluidCalendar`) and choose an expiry period.
    4. Click **Add**.
    5. Immediately copy the **Value** of the generated secret.

    <Warning>
      The client secret value is only shown once. If you navigate away without copying it, you must delete the secret and create a new one.
    </Warning>
  </Step>

  <Step title="Enter credentials in FluidCalendar">
    1. Open FluidCalendar and go to **Settings → System**.
    2. Scroll to the **Outlook Calendar Integration** section.
    3. Enter the following values:
       - **Client ID**: the Application (client) ID from step 1
       - **Client Secret**: the secret value from step 4
       - **Tenant ID**: leave empty to allow any Microsoft account, or enter your specific Azure AD tenant ID to restrict access to one organization
    4. Click **Save**.

    Alternatively, set these environment variables before starting FluidCalendar:

    ```bash .env
    AZURE_AD_CLIENT_ID="your-application-client-id"
    AZURE_AD_CLIENT_SECRET="your-client-secret-value"
    AZURE_AD_TENANT_ID="common"
    ```

    <Info>
      Use `common` as the Tenant ID to allow sign-in from any Microsoft account. Use your specific tenant ID (a GUID) to restrict access to a single Azure AD directory.
    </Info>
  </Step>

  <Step title="Connect your Outlook account">
    1. Go to **Settings → Integrations**.
    2. Click **Connect Outlook Calendar**.
    3. Sign in with your Microsoft account and grant the requested permissions.
    4. After authorization, your Outlook calendars will appear in the calendar sidebar.
  </Step>
</Steps>

## How sync works

FluidCalendar connects to Outlook through the Microsoft Graph API (`https://graph.microsoft.com/v1.0`).

### Calendar sync

- **Outlook → FluidCalendar**: Events from your connected Outlook calendars are fetched and displayed in FluidCalendar. The initial sync retrieves events spanning 5 years in the past through 3 years into the future. Recurring events are fetched as a series master with all individual instances expanded.
- **FluidCalendar → Outlook**: When you create, update, or delete an event in FluidCalendar, the change is pushed to Outlook immediately using the Microsoft Graph API.

For recurring events, you can update or delete a **single instance** or the **entire series**. FluidCalendar handles both by targeting the correct event ID on the Graph API.

### Delta sync (incremental updates)

After the initial sync, FluidCalendar uses the Microsoft Graph delta token to perform incremental syncs. Only events that have changed since the last sync are fetched, which keeps sync fast regardless of how many events you have.

### Outlook Tasks sync

The `Tasks.ReadWrite` permission enables bidirectional sync with Outlook Tasks:

- Tasks created in Outlook appear in FluidCalendar's task list.
- Tasks you create or update in FluidCalendar are written back to Outlook Tasks.

## Managing your calendars

Once connected, each Outlook calendar is listed in the sidebar as a separate feed. You can:

- **Show or hide** a calendar by toggling its checkbox.
- **Manually sync** a calendar by clicking the sync icon next to it.
- **Remove** a calendar by clicking the trash icon.

## Tenant ID explained

The Tenant ID controls which Microsoft accounts can authorize your app:

| Tenant ID value | Who can sign in |
|---|---|
| *(empty or `common`)* | Any Microsoft account (personal or organizational) |
| `organizations` | Only work or school accounts from any Azure AD directory |
| `consumers` | Only personal Microsoft accounts |
| A specific GUID | Only accounts from that single Azure AD tenant |

For most self-hosted FluidCalendar deployments, leaving the Tenant ID empty (which uses `common`) is the right choice.

## Production deployment notes

**Update redirect URIs.** After deploying to production, add your production URL to the **Redirect URIs** in your Azure app registration under **Authentication**.

**Grant admin consent.** If your organization requires admin consent for API permissions, have an Azure administrator grant consent from the **API permissions** page.

**Client secret expiry.** Client secrets expire based on the expiry period you chose. Set a reminder to rotate the secret before it expires — an expired secret will prevent all users from reconnecting their Outlook accounts.

## Troubleshooting

<AccordionGroup>
  <Accordion title="AADSTS50011: redirect URI mismatch">
    The redirect URI used by FluidCalendar does not match any registered URI in your Azure app. Ensure the **Redirect URI** in Azure is exactly `{YOUR_URL}/api/auth/callback/azure-ad` with no trailing slash.
  </Accordion>

  <Accordion title="AADSTS65001: user has not consented">
    The user is being asked to consent to permissions that haven't been pre-approved. Either ask an admin to grant admin consent, or ensure each user consents when connecting their account.
  </Accordion>

  <Accordion title="Token expired or invalid">
    If FluidCalendar loses access after a period of time, the client secret may have expired. Generate a new secret in **Certificates & secrets** and update the value in **Settings → System**.
  </Accordion>

  <Accordion title="Calendars not appearing after connection">
    Confirm that `Calendars.ReadWrite` is listed and granted in **API permissions**. If the permission was added after the user first connected, the user must disconnect and reconnect to receive the updated permission scope.
  </Accordion>
</AccordionGroup>
